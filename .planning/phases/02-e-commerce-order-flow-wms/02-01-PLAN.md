  <action>
    Create the order service following the state machine pattern from research.

    **Order status transitions:**
    ```
    draft -> awaiting_payment (BACS/card)
    draft -> confirmed (credit account -- no payment wait)
    awaiting_payment -> confirmed (payment received)
    awaiting_payment -> cancelled
    confirmed -> allocated (after FIFO allocation)
    allocated -> picking
    picking -> packed
    packed -> dispatched
    dispatched -> delivered
    delivered -> completed
    ```

    **Exports:**
    - `ORDER_TRANSITIONS: Record<OrderStatus, OrderStatus[]>` -- the transition map
    - `canTransition(from, to): boolean` -- validates a transition
    - `transitionOrder(tx, orderId, newStatus, metadata?)` -- performs transition in a transaction, throws if invalid
    - `createOrder(tx, data)` -- creates order + order lines in transaction, generates order number using PostgreSQL-safe sequence (query max order number for current month, increment). Format: `ORD-YYYYMM-NNNNNN`. Returns created order. IMPORTANT: This function must fetch product prices from the products.pricePence column to populate unitPricePence and lineTotalPence on order lines. Never trust client-side prices.
    - `allocateOrderStock(tx, orderId)` -- calls allocateFIFO from batch-service for each order line, transitions order to 'allocated'. This wires the existing FIFO logic into the order flow.

    **Order number generation:** Use the same pattern as generateTorkeBatchId in batch-service.ts -- query max within transaction to prevent races. Format `ORD-{YYYYMM}-{NNNNNN}` (6-digit zero-padded sequence per month).

    **Import allocateFIFO from batch-service.ts** for the allocateOrderStock function. The existing allocateFIFO already handles cross-batch splitting and transaction safety.

    Type imports: use the schema types from orders.ts (OrderStatus from the enum, orders/orderLines tables).

    **src/__tests__/allocation/fifo-allocation.test.ts:** Fill in the test stubs from Wave 0.
    - Test canTransition: valid transitions return true, invalid return false
    - Test ORDER_TRANSITIONS map covers all statuses
    - Test createOrder generates sequential order numbers
    - Test allocateOrderStock calls allocateFIFO per order line (mock batch-service)
