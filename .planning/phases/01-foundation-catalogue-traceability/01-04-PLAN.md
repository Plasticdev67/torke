    1. **Batch Service** (`src/server/services/batch-service.ts`):

    **`generateTorkeBatchId()`:**
    Generate batch IDs in format `TRK-YYYYMMDD-NNNN`. Query the database for the highest NNNN for today's date and increment. Use a database sequence or MAX query within the transaction to prevent race conditions.

    **`completeGoodsIn(data)`:**
    This is the critical function. It runs inside a database transaction:
    ```typescript
    async function completeGoodsIn(data: GoodsInData) {
      return db.transaction(async (tx) => {
        // 1. Create or find supplier
        const supplier = await findOrCreateSupplier(tx, data.supplierName);

        // 2. Create supplier batch record
        const [supplierBatch] = await tx.insert(supplierBatches).values({
          supplierId: supplier.id,
          supplierBatchNumber: data.supplierBatchNumber,
          productId: data.productId,
          quantityReceived: data.quantity,
          manufacturerCertUrl: data.certKey, // R2 key from cert upload
        }).returning();

        // 3. Generate Torke batch ID
        const torkeBatchId = await generateTorkeBatchId(tx);

        // 4. Create batch record with status='available' (goods-in is complete)
        const [batch] = await tx.insert(batches).values({
          torkeBatchId,
          supplierBatchId: supplierBatch.id,
          productId: data.productId,
          receivedBy: data.userId,
          quantity: data.quantity,
          quantityAvailable: data.quantity,
          quantityReserved: 0,
          status: 'available',  // TRACE-05: only available after goods-in complete
          expiryDate: data.expiryDate, // WMS-05: for chemical products
          inspectionNotes: data.inspectionNotes,
        }).returning();

        // 5. Create stock item
        await tx.insert(stockItems).values({
          batchId: batch.id,
          productId: data.productId,
          quantity: data.quantity,
        });

        // 6. Create verification token (TRACE-18, TRACE-19)
        const [token] = await tx.insert(verificationTokens).values({
          batchId: batch.id,
        }).returning();

        return { batch, verificationToken: token.token, torkeBatchId };
      });
    }
    ```

    CRITICAL: The entire operation is a single transaction. If any step fails, nothing is committed. The batch is created with status='available' because goods-in is complete at this point (cert uploaded, batch assigned). Status='pending' would be used if we had a multi-step approval process, but per TRACE-05, completion = availability.

    **`allocateFIFO(tx, productId, quantityNeeded)`:**
    FIFO allocation as specified in research Section 10. Queries available batches ordered by goodsInDate ASC. Allocates from oldest first. If oldest batch has insufficient quantity, splits across batches. Returns array of allocations. Throws if insufficient total stock. This is used in Phase 2 but the logic is defined now (TRACE-06).

    **`recallQuery(supplierBatchNumber)`:**
    Traces the chain: supplierBatch -> batch(es) -> allocation(s) -> orderLine(s). Returns all affected orders for a given supplier batch. This proves the data model works for TRACE-08.

    2. **QR Service** (`src/server/services/qr-service.ts`):
    - `generateQRDataUrl(token)` — returns base64 data URL for browser display and label embedding
    - `generateQRBuffer(token)` — returns PNG buffer for server-side use
    - `generateQRSvg(token)` — returns SVG string for PDF embedding
    - All use `buildVerificationUrl(token)` from constants
    - Error correction level H (survives 30% damage — essential for warehouse labels)
    - QR width 200px

    3. **tRPC batches router** (`src/server/trpc/routers/batches.ts`):
    - `batches.completeGoodsIn` — protected procedure (warehouse role). Validates input with Zod. Calls `completeGoodsIn()`. Returns batch and token.
    - `batches.getById` — get batch details with supplier batch, mill cert, and verification token.
    - `batches.list` — list batches with filters (productId, status, date range). Paginated.
    - `batches.getByToken` — public procedure. Look up batch by verification token UUID. For QR verification page.

    4. **tRPC stock router** (`src/server/trpc/routers/stock.ts`):
    - `stock.overview` — protected procedure. Returns stock levels grouped by product, with batch detail (batch ID, quantity available, quantity reserved, expiry date).
    - `stock.expiringBatches` — protected procedure. Returns batches with expiry dates within N days (default 30). For WMS-05 alerts.

    5. Wire batches and stock routers into appRouter.

    Server Component. PUBLIC — no authentication required. This is the page people see when scanning QR codes on product labels.

    **Layout:**
    - Torke logo at top
    - "Product Verification" heading
    - Product information card: name, SKU, category
    - Batch information card: Torke batch ID, goods-in date, quantity in batch
    - Supplier information card: supplier name, supplier batch number, production date

    **TRACE-19 compliance:** The token is an opaque UUID. The batch ID (TRK-YYYYMMDD-NNNN) is shown on the verification page but is NOT in the URL. The URL reveals nothing about the batch until it is loaded from the database.
